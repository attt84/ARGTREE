プロジェクト概要

国会議事録をAIで構造化し「議論木」として可視化するWebアプリケーションの、現状把握と論点整理のための中核ドキュメント。

フェーズ位置づけ（2026-07-16 更新）
「無料APIのみ・コストゼロ」制約下のMVPを経て、同日中に本格実装フェーズのコア（ハイブリッドGraphRAG）を実装完了。中核課題「過去の関連事象を1つずつ手繰っていく」マルチホップ推論に対応する深掘り追跡モードが加わった。設計と実装状況はマルチホップ検索構想、更新後の構成はREADMEを参照。フェーズ詳細ページ（A〜G）の「現状」記述はMVP時点の記録として残している。

何を作っているか

国会議論木は、キーワード（例:「AI規制」）を入力すると、国立国会図書館の国会会議録検索APIから関連発言を取得し、Gemini（gemini-3.5-flash）で「大テーマ・賛成・反対・中立・解決策」のノードに分類した議論木（Argument Tree）を描画するアプリケーション。各ノードはクリックで深掘りでき、実際の発言引用と議事録の一次情報URLつきの解説をAIが生成する。

利用フローは次の5段階:

  1. キーワード入力（オプションで「超検索」= AIによるクエリ拡張をON）
  2. 議事録の取得とGeminiによる構造化 → ツリー描画（緑=賛成、赤=反対、黄=解決策、青=テーマ）
  3. 気になるノードをクリック → サイドバー展開
  4. 「AIで深掘り解説を生成」→ 背景・引用・対立意見のレポート生成
  5. 解説内の一次情報URL（国会会議録の発言単位リンク）でファクトチェック

A クエリ拡張（超検索）

国会APIが「スペース区切り＝AND検索」しかサポートしない制約を、LLMによるQuery Expansionで乗り越える機能。

目的

単一キーワードでは取りこぼす「周辺議論」（例:「AI規制」に対する「人工知能 ガバナンス」「ディープフェイク」）を、AIが生成した関連クエリで網羅的に取得し、疑似的なセマンティック検索を実現する。

現状

  • expand_search_query() がGeminiに関連クエリを生成させ、JSON配列を手動パース。最大3クエリに切り詰めて返す。
  • プロンプトには出力例（「少子化対策」→ ["少子化対策", "児童手当", "育休 支援"]）を含み、1クエリ=1〜2語に制限。
  • 生成失敗・パース失敗時は元クエリ1件にフォールバック（ユーザーには失敗が見えない）。
  • 拡張されたクエリはレスポンスに含まれず、ユーザーはどんなクエリで検索されたか知る術がない（議事録テキスト中の [クエリ: q] ラベルがLLM経由で見える可能性はあるが保証なし）。

設計案

  • JSONパースは C と同じく structured output（response_schema）へ寄せる（O1/O3と同時実施）。
  • ArgumentResponse に expanded_queries: list[str] を追加し、フロントでチップ表示して透明性を確保する。

論点

#論点選択肢判断材料・方針状態
A-1拡張クエリの可視化①レスポンスに含めUI表示 ②現状のまま非表示検索の透明性・デバッグ性向上 vs UIの複雑化。実装コストは小さい。推奨: ①未着手
A-2拡張数3固定の妥当性①3固定 ②2〜5で可変（UI指定 or 自動）クエリ数×国会APIレイテンシ（逐次処理なら線形増加）。O6の並列化が入れば増やす余地。推奨: 並列化後に再検討未着手

B 議事録取得（国会API）

検索キーワードに関連する発言を一次情報URL付きで収集する、RAGの「Retrieval」部分。

目的

ユーザーのクエリ（または拡張クエリ群）に該当する国会発言を、後段のLLM処理に適した密度・形式で取得する。開発初期に「会議単位取得」から「発言単位取得」へピボット済み（決着済み論点 R1）。

現状

取得方式
requests による同期・逐次リクエスト（タイムアウト10秒）
取得件数
クエリごとに max(5, 15 ÷ クエリ数) 件（単独=15件、超検索3クエリ=各5件）
抽出フィールド
会議名・日付・発言者・発言本文・発言URL の5種のみ
整形
「--- 会議名 (日付) [クエリ: q] --- / URL / [発言者]: 本文」の形式で1本のテキストに連結
エラー処理
クエリ単位の失敗はログしてスキップ。全滅時は「関連する議事録が見つかりませんでした。」という文字列を返し、それがそのままLLMに渡る

設計案

  • httpx.AsyncClient + asyncio.gather による並列取得（docs/technical_decisions.md でも既知の課題として明記済み）。
  • 連結済み文字列ではなく発言のリスト（構造化データ）として保持し、重複排除・キャッシュ・件数制御をしやすくする。プロンプト用テキストへの整形は最後に行う。
  • 超検索時に同一発言が複数クエリでヒットした場合の重複排除（speechIDまたはspeechURLをキーに）。

論点

B-3重複発言の排除①speechIDでdedupe ②なし（現状）超検索時、類似クエリが同じ発言を重複取得するとコンテキストを浪費する。実測（EDA）で重複率を確認してから未着手

C 議論木生成（Gemini構造化）

議事録テキストを nodes / edges のグラフJSONへ変換する、本アプリの心臓部。

目的

数万字の議事録抜粋から「大テーマ（theme）を1つのルートとし、賛成（pro）・反対（con）・中立（neutral）・解決策（solution）が派生するツリー」を安定して生成する。ラベルは20文字以内、発言者名を source に付与。

現状

  • プロンプトで「JSONのみ出力・コードブロック不要」を指示し、返答の ```json フェンスを手動で剥がして json.loads → Pydantic（ArgumentTree）で検証。
  • 議事録は先頭30,000字で切断してプロンプトに注入（後半の発言は黙って捨てられる）。
  • 失敗時は (1) カレントディレクトリに gemini_error.txt を書き出し、(2) 「解析エラー」ノードを持つダミーツリーをHTTP 200で返却。フロントは正常なツリーとして描画するため、ユーザーはエラーと気づきにくい。
  • SDKは非推奨の google-generativeai。生成設定（temperature等）は未指定でモデル既定値に依存。

設計案

  • google-genai SDKへ移行し、response_schema にPydanticモデルを直接渡す構造化出力へ変更 → フェンス剥がし・手動パースを全廃し、パース失敗そのものをほぼ根絶する。
  • 失敗時はダミーツリーではなく明示的なエラー応答（例: 502 + 「AI解析に失敗しました。再試行してください」）を返し、フロントに再試行UIを設ける。
  • gemini_error.txt の書き出しを廃止し、logger に一本化する。

D ツリー可視化（React Flow）

生成されたグラフJSONを、賛否が一目でわかる自動レイアウト済みツリーとして描画する。

目的

「どの意見がどの意見への反論・解決策なのか」という構造的関係を視覚的に伝える。タイプ別の色分け（テーマ=青 / 賛成=緑 / 反対=赤 / 解決策=黄 / 中立=白）で議論の勢力図を直感的に把握できるようにする。

現状

  • React Flow（@xyflow/react v12）+ dagre で上下方向（TB）の自動レイアウト。ノード幅250px・高さ80px想定。
  • dagre をコンポーネント内で require('dagre') により動的読込（ESM環境でのハック）。失敗時は3列グリッド配置にフォールバック。

E 深掘り解説（RAG）

ノードの論点について、背景・実際の発言引用・一次情報URL・対立意見をAIが解説する、本アプリの信頼性の要。

目的

「短い要約だけでは何も学べない・ファクトチェックできない」という課題（開発経緯に記録済み）への回答。解説には必ず発言引用と国会会議録の一次情報URLを含めることで、学習・調査ツールとしての信頼性を担保する。

現状

  • POST /api/node_detail が { query, node_label } を受け、議事録を元クエリのみで再取得してから generate_node_detail() でMarkdown解説を生成。
  • プロンプトの要求事項: ①背景2〜3段落 ②関連発言の引用+URL必須 ③対立意見・課題の整理。
  • 生成失敗時は「申し訳ありません…」という固定文字列をHTTP 200で返す。
  • キャッシュはなく、ノードをクリックして生成するたびに国会API再取得+LLM呼び出しが走る。
構造上の問題（R-3 / O2）
超検索でツリーを作った場合、ツリーは「拡張3クエリ分の議事録」から生成されるが、深掘りは「元クエリ単独の議事録」で行われる。ノードによってはその論点の根拠となった発言が深掘り時のコンテキストに存在しないことがあり、引用の欠落や別文脈での解説（実質ハルシネーション）につながる。

設計案

  • 検索時に取得した議事録コンテキストをサーバ側に短期キャッシュ（メモリ+TTL）し、レスポンスに search_id を含める。深掘りは search_id で同一コンテキストを参照する。キャッシュ失効時のみ再取得にフォールバック。
  • 副次効果: 深掘りのレイテンシ短縮（国会API往復の削減）と外部API負荷の低減。

論点

#論点選択肢判断材料・方針状態
E-1コンテキスト整合とキャッシュ（O2）①search_id+サーバ側キャッシュ ②拡張クエリをフロント経由で引き回す ③現状①はAPI契約の変更が小さく整合性を根本解決。②はキャッシュなしで整合のみ解決（レイテンシ改善なし）。推奨: ①要判断

マルチホップ検索構想（GraphRAG）

本格実装フェーズの中核課題。「過去の関連事象を1つずつ手繰っていく」推論に、検索アーキテクチャがどう応えるかの設計議論。

課題の定義: マルチホップ推論

国会の議論は孤立して生まれない。「なぜこの法案にこの条項があるのか」に答えるには、直近の審議 → それを促した数年前の事件・不祥事 → その際の附帯決議 → さらに前の法改正…と、関連事象を時系列・因果の鎖に沿って1つずつ遡る必要がある。これが本プロジェクトが本当に解きたい検索課題である。

なぜベクトル検索だけでは深部に届かないか
  • 類似度の減衰: ホップごとに語彙（事件名・法案名・人名）が入れ替わるため、元クエリとの埋め込み類似度は急速に下がる。2ホップ目以降の文書は「元の質問に似ていない」。
  • チャンクの独立性: 発言単位の索引は発言間の参照関係を持たず、鎖が索引の段階で切れている。
  • 参照の暗黙性: 議事録の言及は「先般の事案」「昨年の改正」のような指示表現が多く、埋め込みにも全文検索にも乗りにくい。
現行MVPの検索（1ホップAND検索＋クエリ拡張）は「横の網羅」を広げる仕組みであり、「縦の深さ」には潜れない。

アーキテクチャ選択肢の比較

案概要マルチホップ適性構築・運用コスト評価
1. ベクトルRAG単独コーパスを埋め込み、類似検索のみで取得低（上記の減衰問題）低〜中入口の発見には有効。単独では課題に届かない
2. 反復・エージェント型検索LLMが「検索 → 精読 → 次の手がかり（固有名詞・日付）を抽出 → 再検索」を繰り返す（IRCoT / ReAct系）中〜高構築コスト低（索引不要）／オンラインコスト・レイテンシ大既存APIの上でも動く即効策。ただし各ホップが全文検索頼みだと言い換え・暗黙参照に弱い
3. GraphRAG（知識グラフ＋トラバーサル）【提案中】事象・議案・人物・発言をノード、審議・言及・因果をエッジとする知識グラフを事前構築し、エッジを辿って深部へ到達する高（ホップ＝エッジ辿りとなり、到達性が構造的に担保される）高（構築時のLLM抽出）課題との適合が最も良い。グラフ構築の品質が成否を握る
4. Microsoft型GraphRAG（コミュニティ要約）グラフのコミュニティ検出＋階層要約で「全体像を問う質問」に回答中（俯瞰向き）高チェーン追跡より「テーマの全体像」向き。議論木生成の品質向上として部分採用の価値あり
5. ハイブリッド【推奨叩き台】ベクトル＝入口の発見、グラフ＝ホップの追跡、エージェント＝経路の判断という三層合成高高各案の弱点を相互補完し、段階的に構築できる

国会データが最初から持つグラフ構造（抽出コスト不要の資産）

GraphRAG構築コストの大半はLLMによる関係抽出だが、国会議事録には抽出する前から使える構造が多い。これを土台にすれば、グラフ構築を低リスクで段階化できる。

階層エッジの例構築方法
① メタデータ発言 → 会議 → 委員会 → 国会回次 ／ 発言者 → 会派 ／ 同一議題の継続審議のつながりAPIレスポンスのフィールドから機械的に構築（LLM不要）
② 明示参照発言中の法案名・法律名・事件名への言及 → 議案・事象ノードへのエッジ固有表現抽出（ルール＋LLM併用）。衆参の議案DB・日本法令索引との突合も視野
③ 暗黙参照・因果「先般の事案を受けて」→ 事象ノードへの解決 ／ 事件 → 法改正の因果エッジコンテキスト付きLLM抽出。最も高コストだが最も高価値（縦の手繰りの本体）

X-1. SDK移行: google-generativeai → google-genai（O1）

  • 事実関係: google-generativeai（現在使用中）はサポートが2025-08-31に終了済み。Googleは統一SDK google-genai への移行を公式に強く推奨している（2026-07-16に公式情報で確認）。
  • 移行のメリット: ①サポート復帰 ②response_schema によるPydantic直結の構造化出力（O3） ③async対応（O6のエンドポイントasync化と整合） ④新モデル・新機能への追随。
  • 影響範囲: requirements.txt 1行 + api_gemini.py の3関数。書き換えイメージ:
# Before (google-generativeai / 非推奨)
import google.generativeai as genai
genai.configure(api_key=api_key)
model = genai.GenerativeModel('gemini-3.5-flash')
response = model.generate_content(prompt)

# After (google-genai / 推奨・構造化出力つき)
from google import genai
client = genai.Client(api_key=api_key)
response = client.models.generate_content(
    model='gemini-3.5-flash',
    contents=prompt,
    config={
        'response_mime_type': 'application/json',
        'response_schema': ArgumentTree,   # Pydanticモデルを直接指定
    },
)
tree = response.parsed  # パース済みオブジェクト

X-2. デプロイ戦略（O9）

案構成利点難点
案1Vercel（front）+ Cloud Run（back）Next.jsとの親和性最高。バックエンドはコンテナで自由度が高い。無料枠で開始可能2サービスの管理。CORS・環境変数の整備（O5必須）
案2単一VPS（front+backを同居）構成が単純。固定費のみ運用（TLS・更新）が全部自前
案3Hugging Face Spaces等のPaaS一体型デモ公開が最速Next.jsフロントの本格ホスティングには不向き

ポートフォリオ公開が目的なら案1が推奨。ただし公開時はGemini APIコスト・国会APIへの負荷・悪用対策（レート制限）の検討が必要。

X-3. レート制限・コスト

  • LLM呼び出し回数: 通常検索=1回、超検索=2回（拡張+構造化）、深掘り=1回/ノード。
  • Gemini無料枠のRPM/RPD制限に達すると現状は生のエラーが露出する（F-1）。Exponential Backoffつきリトライ（F-4）と、429時の「混雑中」メッセージが必要。
  • 国会APIは公共インフラであり、並列化（O6）実施時もクエリ数・頻度は節度を保つ（同時3リクエスト程度なら問題ない想定）。
