# 技術的アーキテクチャと論点 (Technical Decisions)

本ドキュメントでは、`argdrasill` において採用した技術的な意思決定（Architecture Decision Records）と、その選定理由、および今後のスケーラビリティに向けた技術的論点について解説します。

## 1. 全体アーキテクチャ設計

### 1.1 バックエンド: FastAPI (Python) の採用
- **理由**:
  - LLM（Google Generative AI SDK）やデータ処理系ライブラリとの親和性が圧倒的に高いため。
  - Pydanticを用いた型安全なリクエスト/レスポンスのバリデーションが強力であり、LLMに要求するJSONスキーマの定義（`ArgumentTree`, `ArgumentNode`）とバックエンドのモデル定義を一元化できるため。
  - 非同期処理（`async def`）がネイティブサポートされており、将来的に複数APIの並列呼び出し（I/Oバウンドな処理）を最適化しやすいため。

### 1.2 フロントエンド: Next.js (App Router) + React Flow
- **理由**:
  - Reactエコシステムにおいて、複雑なノードベースのグラフ描画を行うためのデファクトスタンダードである `React Flow (@xyflow/react)` を利用するため。
  - Tailwind CSSと組み合わせることで、カスタムノード（賛成・反対などの動的スタイリング）を非常に高速に実装できました。

## 2. RAG (Retrieval-Augmented Generation) パターンの実装

本システムは、典型的な「ベクトルDBを用いたRAG」ではなく、「外部APIの動的検索を利用したオンザフライ型RAG」を採用しています。

### 2.1 動的プロンプト・インジェクション
- ユーザーの検索クエリに応じて国会図書館APIを叩き、取得した生のテキストデータを即座にプロンプト（コンテキスト）としてGeminiに注入します。
- **論点と対応**:
  - *プロンプトインジェクション攻撃への耐性*: 現在は国会の公的議事録のみをコンテキストとして挿入しているため、外部データを通じた悪意あるプロンプト操作のリスクは低いと判断しています。

### 2.2 LLMの構造化出力 (Structured Output)
- Geminiに対して、純粋なJSONフォーマットのみを出力するよう厳密にプロンプトエンジニアリングを行っています。
- バックエンド（FastAPI側）で `json.loads` を行い、失敗した場合はフォールバックとして「解析エラー」ノードを持つダミーツリーを返すことで、フロントエンドのクラッシュを防ぐフォールトトレラントな設計としています。

## 3. 「超検索」機能におけるアーキテクチャ的工夫

国会図書館APIの「AND検索のみ」という仕様上の制約を打破するため、「Query Expansion（クエリ拡張）」を実装しました。

### 3.1 実装フロー
1. **Pre-processing (LLM)**: ユーザー入力（例「AI」）をGeminiに渡し、関連キーワードのJSON配列（`["AI規制", "人工知能", "機械学習"]`）を生成させる。
2. **Parallel Fetching**: 生成された複数キーワードに対して、`api_kokkai.py` にて並列的（シーケンシャル）にリクエストを送信。
3. **Context Merging**: 取得した複数パターンの議事録データを1つの巨大なテキストブロックに結合し、メインのプロンプトへ注入。

### 3.2 今後の技術的課題（スケーラビリティ）
- **同期処理から非同期処理への移行**: 現在の超検索（複数クエリの発行）は `requests` ライブラリを用いた同期処理で実装されています。より高速なレスポンスを実現するためには、`httpx` などを導入し `asyncio.gather` を用いた並列フェッチへのリファクタリングが推奨されます。
- **レートリミット対策**: Gemini APIの無料枠（リクエスト数制限）に達した場合のエラーハンドリングは実装済みですが、実稼働環境では Exponential Backoff（指数的バックオフ）を伴うリトライ制御の導入が必要です。
- **ベクトル検索への移行**: 今後データ量が膨大になった場合、バッチ処理で議事録を事前にEmbedding化し、Vector DB（Pinecone等）に保存する本格的なRAGアーキテクチャへの移行も視野に入れています。
